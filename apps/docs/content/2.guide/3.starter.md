`apps/starter` is a minimal, clonable Nuxt 4 admin app built on AdminLTE 4. It `extends` the [`@adminlte/dashboard`](./routing) layer — so the dashboard shell, layouts, Pinia auth store, route middleware and a mock auth API come for free — and adds its own brand, navigation, pages and data on top. Use it as the starting point for a real project instead of wiring the layer up from scratch.

## What it includes

| Area | Where |
| --- | --- |
| Auth suite (login / register / forgot / reset / lockscreen) + RBAC | inherited from `@adminlte/dashboard` |
| Users CRUD with server-side search / sort / paginate + validation | `app/pages/users`, `server/api/users` |
| Live Analytics / Calendar / Board (mock data over `useFetch`) | `app/pages/{analytics,calendar,board}`, `server/api` |
| Toasts, theme customizer, route transitions | inherited from the layer |
| i18n (en / es, cookie-detected, no URL prefix) | `i18n/locales/*.json` |

The bundled mock user directory lets you sign in immediately with **admin@example.com** / **password** (an admin) or **editor@example.com** / **password** (an editor).

## Clone and run

The starter lives in this monorepo. Run it directly:

```bash [Terminal]
pnpm install
pnpm --filter adminlte-starter dev      # http://localhost:3000
```

To start your own project, copy `apps/starter` out of the repo and replace the `workspace:*` dependency on `@adminlte/dashboard` (in `package.json`) with the published version.

## How it extends the layer

Everything dashboard-related is inherited through Nuxt layer `extends`. The app's `nuxt.config.ts` is intentionally thin:

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  compatibilityDate: '2025-05-01',
  extends: ['@adminlte/dashboard'],
  modules: ['@nuxtjs/i18n', '@nuxt/eslint'],
  i18n: {
    defaultLocale: 'en',
    strategy: 'no_prefix', // locale via cookie, no URL prefix
    locales: [
      { code: 'en', name: 'English', file: 'en.json' },
      { code: 'es', name: 'Español', file: 'es.json' },
    ],
    detectBrowserLanguage: { useCookie: true, cookieKey: 'i18n_locale', redirectOn: 'root' },
  },
  runtimeConfig: { public: { appName: 'AdminLTE Starter' } },
})
```

### Branding and navigation

Rebrand and define the sidebar menu in `app/app.config.ts`. Nuxt deep-merges this over the layer's defaults, so you only specify the keys you want to change. Menu `text` values are i18n keys (they fall back to the literal key when i18n is absent), and items can be restricted to a role with `roles`:

```ts [app/app.config.ts]
export default defineAppConfig({
  adminlte: {
    brand: { logo: '/assets/img/AdminLTELogo.png', text: 'AdminLTE', href: '/' },
    menu: [
      { type: 'item', text: 'nav.dashboard', href: '/', icon: 'bi-speedometer2' },
      { type: 'header', text: 'nav.features' },
      { type: 'item', text: 'nav.analytics', href: '/analytics', icon: 'bi-graph-up' },
      // admin-only entry — hidden for non-admins:
      { type: 'item', text: 'nav.users', href: '/users', icon: 'bi-people', roles: ['admin'] },
    ],
    notifications: [
      { text: 'New sale recorded', icon: 'bi-bag-check', time: '3 mins' },
    ],
  },
})
```

## Auth and RBAC

Auth is backed by the layer's cookie-backed Pinia store, `useAuthStore()`. The token cookie is SSR-readable, so guards and the topbar resolve the signed-in user on the server too.

| Member | Type | Description |
| --- | --- | --- |
| `isAuthenticated` | `ComputedRef<boolean>` | True when a token cookie is set. |
| `user` | `Ref<AuthUser \| null>` | The hydrated user. |
| `roles` | `ComputedRef<string[]>` | Role keys for the current user. |
| `isAdmin` | `ComputedRef<boolean>` | Shortcut for `roles.includes('admin')`. |
| `hasRole(role)` | `(role: string) => boolean` | Whether the user has a single role. |
| `hasAnyRole(required)` | `(required: string[]) => boolean` | True if `required` is empty or the user has at least one of them. |
| `login(email, password)` | `Promise<AuthUser>` | POSTs `/api/auth/login`, stores token + user. |
| `register(payload)` | `Promise<AuthUser>` | POSTs `/api/auth/register`. |
| `fetchUser()` | `Promise<AuthUser \| null>` | Hydrates the user from the token. |
| `logout()` | `() => void` | Clears the token and user. |

Guard a page (and optionally restrict it by role) with `definePageMeta`. The named `auth` middleware redirects unauthenticated visitors to `/login?redirect=…`, and throws a 403 when the page declares `roles` the user does not hold:

```vue [app/pages/users/index.vue]
<script setup lang="ts">
// any signed-in user:
definePageMeta({ middleware: 'auth' })
// admins only:
definePageMeta({ middleware: 'auth', roles: ['admin'] })
</script>
```

## Users CRUD (server-side search / sort / paginate)

The Users section is the full-stack reference. The page debounces the search box, then drives a single reactive `useFetch` — every change to the query (search term, sort column/direction, page) re-runs the request, so filtering and paging happen on the server:

```ts
const query = computed(() => ({ q: q.value, sort: sort.value, dir: dir.value, page: page.value, size: size.value }))
const { data, pending, refresh } = await useFetch('/api/users', { query })
```

The matching endpoint reads the query and delegates to the in-memory `users-db` helper, which returns `{ rows, total, page, size }`:

```ts [server/api/users/index.get.ts]
export default defineEventHandler((event) => {
  const q = getQuery(event)
  return listUsers({
    q: typeof q.q === 'string' ? q.q : '',
    sort: (q.sort as keyof User) || 'id',
    dir: q.dir === 'desc' ? 'desc' : 'asc',
    page: Number(q.page) || 1,
    size: Number(q.size) || 10,
  })
})
```

Create / read / update / delete are the sibling route files (`index.post.ts`, `[id].get.ts`, `[id].put.ts`, `[id].delete.ts`), with input checked by `server/utils/validate-user.ts`. Delete mutations call `useToast()` for feedback (`success`, `error`, `warning`, `info`).

## Live Analytics, Calendar and Board

Three feature pages demonstrate live, SSR-rendered data fed by mock Nitro endpoints — swap the endpoints for your backend and the pages keep working:

- **Analytics** (`/analytics`) — `useFetch('/api/analytics')` feeding ApexCharts widgets.
- **Calendar** (`/calendar`) — FullCalendar reading `GET /api/events` and posting new ones to `POST /api/events`.
- **Board** (`/board`) — a SortableJS kanban backed by `GET /api/board` and `POST /api/board/move`.

The dashboard home (`/`) pulls its widgets from `server/api/dashboard/stats` and `server/api/dashboard/activity`.

## i18n

The app ships English and Spanish translations under `i18n/locales/*.json`. With `strategy: 'no_prefix'`, the active locale is stored in the `i18n_locale` cookie (no URL prefix), and the layer's language switcher in the topbar changes it. Sidebar menu labels and page titles are looked up via `useI18n().t(...)`.

## Swapping the mocks for a real backend

All data is mocked in-process and is meant to be replaced. Each layer below is independent — the route handlers depend only on the helper modules, so you can swap one without touching the pages:

| Mock | Replace with |
| --- | --- |
| `@adminlte/dashboard` `server/api/auth/*` + `server/utils/auth-mock.ts` | Your real login / register / session endpoints (DB lookup + JWT/session). |
| `server/utils/users-db.ts` | A real database; keep the `listUsers` / `getUser` / `createUser` / `updateUser` / `removeUser` signatures and the route handlers stay unchanged. |
| `server/utils/board-db.ts`, `server/utils/events-db.ts` | Your board / calendar persistence. |

Put secrets in `.env` (see `.env.example`); public values must be exposed through `runtimeConfig.public` (e.g. `NUXT_PUBLIC_APP_NAME`).

## Build and deploy

The build output is a standard Nitro server (`node-server` preset by default):

```bash [Terminal]
pnpm --filter adminlte-starter build
node apps/starter/.output/server/index.mjs   # PORT=3000 HOST=0.0.0.0
```

Vercel, Netlify and Cloudflare are auto-detected by their Nuxt support; otherwise set a preset in `nuxt.config.ts` (`nitro: { preset: 'vercel' }`, `'netlify'`, `'cloudflare-pages'`, …). A `Dockerfile` is included for containerized deploys — build it from the repo root.
