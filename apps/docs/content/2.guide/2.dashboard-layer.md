`@adminlte/dashboard` is an AdminLTE 4 starter packaged as a [Nuxt Layer](https://nuxt.com/docs/getting-started/layers). Extending it gives you a working admin shell — the AdminLTE module (components, composables, CSS, SSR-safe theme script), Pinia, default dashboard + auth layouts, a cookie-backed auth store with route middleware, and mock `/api/auth/*` endpoints — that you rebrand and configure entirely from your own `app.config.ts`.

## Usage

Install the layer alongside Pinia, then extend it from your `nuxt.config.ts`.

```bash
pnpm add @adminlte/dashboard @pinia/nuxt pinia
```

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  extends: ['@adminlte/dashboard'],
})
```

The layer already registers `@adminlte/nuxt` and `@pinia/nuxt` and sets sensible AdminLTE defaults (`fixedHeader`, `fixedSidebar`, `sidebarTheme: 'dark'`, `initialColorMode: 'auto'`, `enableSidebarPersistence: true`). You inherit the default dashboard layout, so a page only needs content:

```vue [app/pages/index.vue]
<template>
  <LteAppContent title="Dashboard">Welcome.</LteAppContent>
</template>
```

## Configuring the `adminlte` app config

The default layout (`layouts/default.vue`) is driven by the `adminlte` block of your `app.config.ts`. Nuxt deep-merges app configs across layers, so you only override what you need. The layer ships an empty `menu` default on purpose (Nuxt concatenates arrays when merging), so your menu is the only one rendered.

```ts [app/app.config.ts]
export default defineAppConfig({
  adminlte: {
    brand: { text: 'Acme Admin', logo: '/logo.png', href: '/' },
    user: { name: 'Guest', role: 'Member' },
    menu: [
      { type: 'item', text: 'Dashboard', href: '/', icon: 'bi-speedometer2' },
      {
        type: 'group',
        text: 'Admin',
        icon: 'bi-shield-lock',
        roles: ['admin'],
        children: [
          { type: 'item', text: 'Users', href: '/admin/users' },
        ],
      },
    ],
  },
})
```

### `AdminlteAppConfig`

| Name | Type | Description |
|---|---|---|
| `brand` | `AdminlteBrand` | Sidebar / topbar branding. |
| `user` | `AdminlteUser` | Fallback topbar user (the auth store's user overrides it at runtime). |
| `menu` | `AppMenuNode[]` | Sidebar + command-palette menu (RBAC-filtered). |
| `messages` | `NavMessage[]` | Topbar messages dropdown (rendered only when non-empty). |
| `notifications` | `NavNotification[]` | Topbar notifications dropdown (rendered only when non-empty). |

### `AdminlteBrand`

| Name | Type | Description |
|---|---|---|
| `logo` | `string` | Logo image URL. |
| `text` | `string` | Brand text. |
| `href` | `string` | Brand link target. |

### `AdminlteUser`

| Name | Type | Description |
|---|---|---|
| `name` | `string` | Display name (required). |
| `email` | `string` | Email address. |
| `image` | `string` | Avatar image URL. |
| `role` | `string` | Human-readable role label shown in the topbar. |
| `memberSince` | `string` | "Member since" line in the user menu. |

### RBAC menu filtering with `AppMenuNode`

`AppMenuNode` is the library's `MenuNode` plus an optional `roles?: string[]`. The default layout filters the menu against the signed-in user's roles: a node with `roles` is dropped unless the user matches one (via `auth.hasAnyRole`), and groups that end up with no visible children are removed. Nodes without `roles` are always shown. Labels are also passed through i18n (`text` may be a translation key, falling back to the literal string).

## Layouts

Two layouts ship with the layer.

| Layout | Purpose |
|---|---|
| `default` | The dashboard shell — wires `LteDashboardLayout` to your `app.config.ts` (menu, brand, user) and the auth store (signed-in user + sign-out). The default for every page. |
| `auth` | A bare centered shell that just renders `<slot />` — for login / register / lockscreen pages. AdminLTE's CSS centers the `.login-box` / `.register-box`. |

The default layout reads the current user from the auth store (falling back to `adminlte.user`), passes `route.path` as `current-path`, uses `NuxtLink` as the link component, and on `@logout` calls `/api/auth/logout`, clears the store, and redirects to `/login`. Opt a page into the auth layout with `definePageMeta`:

```vue [app/pages/login.vue]
<script setup lang="ts">
definePageMeta({ layout: 'auth' })
</script>
```

## The auth store

`useAuthStore()` is a Pinia store backed by a cookie, so guards and the topbar can resolve the signed-in user during SSR. The token lives in the `auth-token` cookie (`sameSite: 'lax'`, 7-day `maxAge`); the user is held in `useState('auth:user')`.

```vue
<script setup lang="ts">
const auth = useAuthStore()

async function signIn() {
  await auth.login('admin@example.com', 'password')
}
</script>
```

### State and getters

| Name | Type | Description |
|---|---|---|
| `token` | `string \| null` | Auth token (cookie ref). |
| `user` | `AuthUser \| null` | Signed-in user. |
| `isAuthenticated` | `boolean` | `true` when a token is present. |
| `roles` | `string[]` | The user's role keys (`user.roles`), or `[]`. |
| `isAdmin` | `boolean` | `true` when `roles` includes `'admin'`. |

### Methods

| Name | Signature | Description |
|---|---|---|
| `hasRole` | `(role: string) => boolean` | Whether the user has the given role. |
| `hasAnyRole` | `(required: string[]) => boolean` | `true` if `required` is empty or the user has any of the roles. |
| `login` | `(email, password) => Promise<AuthUser>` | POSTs `/api/auth/login`; stores token + user. |
| `register` | `(payload: { name; email; password }) => Promise<AuthUser>` | POSTs `/api/auth/register`; stores token + user. |
| `fetchUser` | `() => Promise<AuthUser \| null>` | Hydrates the user from the token via `/api/auth/me` (no-op if already loaded; logs out on failure). |
| `logout` | `() => void` | Clears the token and user (client state only). |

`AuthUser` has `name`, `email`, optional `image`, optional `role` (label), and optional `roles` (the keys used for RBAC checks, e.g. `['admin']`).

## Route middleware and RBAC

`middleware/auth.ts` is a named middleware. A page opts in with `definePageMeta`; unauthenticated visitors are redirected to `/login?redirect=<intended path>`. When a page declares `roles`, the middleware enforces them with `hasAnyRole` and throws a **403** (`createError`) if the user doesn't match.

```vue [app/pages/admin/users.vue]
<script setup lang="ts">
definePageMeta({
  middleware: 'auth',
  roles: ['admin'],
})
const auth = useAuthStore()
</script>
```

An empty or omitted `roles` array passes any authenticated user. Pair page-level `roles` with the matching `roles` on your `AppMenuNode` entries so the sidebar hides links the user cannot open.

## Mock auth API

The layer ships mock endpoints under `server/api/auth/*` so auth works out of the box. Replace them with your real backend — the store and middleware only depend on their shapes. The mock directory has two seeded users; both use the password `password`:

- `admin@example.com` → roles `['admin']` (Administrator)
- `editor@example.com` → roles `['editor']` (Editor)

| Endpoint | Method | Body / Headers | Returns |
|---|---|---|---|
| `/api/auth/login` | POST | `{ email, password }` | `{ token, user }` — 401 on bad credentials |
| `/api/auth/register` | POST | `{ name, email, password }` | `{ token, user }` (new user, roles `['user']`) — 422 if fields missing, 409 if email taken |
| `/api/auth/me` | GET | `Authorization: Bearer <token>` | the user — 401 if the token is invalid |
| `/api/auth/logout` | POST | — | `{ ok: true }` |
| `/api/auth/forgot-password` | POST | `{ email }` | `{ ok: true }` — 422 if email missing |
| `/api/auth/reset-password` | POST | `{ token, password }` | `{ ok: true }` — 422 if token or password missing |

Tokens are opaque mock strings (`mock.<base64url email>`); a real implementation would issue and verify a JWT or session.
